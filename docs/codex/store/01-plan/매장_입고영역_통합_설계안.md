# 매장 입고영역 통합 설계안

## 1. 문서 목적

본 문서는 Stockit 기획서 v2.0 기준으로 매장 입고 기능의 목표, 화면 구조, 상태 모델, 데이터 모델, 메뉴 구조, 발주/재고 연계 방식을 정리한 설계 문서다.

매장 입고는 기존 매장 발주의 후속 단계로 정의하며, 승인 완료된 발주가 배송 준비중/배송 중/배송 완료를 거쳐 매장에서 최종 입고 확정되면 재고에 반영되는 흐름을 기준으로 한다.

## 2. 범위와 기본 방향

### 2.1 업무 범위

- 승인 완료된 발주의 입고 진행 상태 조회
- 입고 리스트 조회
- 입고 상세 조회
- 입고 확정 처리
- 입고 완료 이력 조회
- 입고 분석 조회

### 2.2 이번 버전 구현 범위

- 부분 입고, 초과 입고, 부족 입고는 다루지 않는다.
- 입고 확정 단위는 발주 1건 전체다.
- 입고 확정 시 발주 라인 전체 수량을 매장 재고에 반영한다.
- 입고 완료 건은 별도 메뉴가 아니라 입고 리스트 안의 `입고 완료` 상태로 통합 조회한다.
- 창고 실제 출고 UI는 구현 범위에서 제외하되, 상태 전환 로직은 선반영한다.

### 2.3 설계 원칙

- 발주와 입고는 별도 엔티티로 분리하지 않고 `storeOrders`를 source of truth로 사용한다.
- 발주 상태와 입고 상태를 분리해 상위 라이프사이클과 물류 진행 상태를 구분한다.
- 입고 확정 이후 발주 상태는 종료 의미의 `COMPLETED`로 닫는다.
- 입고 확정은 상세에서만 수행하고, 리스트는 처리 대상을 빠르게 식별하는 역할을 담당한다.

## 3. 메뉴 구조와 라우트

### 3.1 메뉴 구조

- 상위 메뉴: `입고 관리`
- 하위 메뉴: `입고 리스트`
- 하위 메뉴: `입고 분석`

기존 `입고 내역` 메뉴는 유지하지 않고, `입고 리스트` 안에서 완료 건까지 함께 조회한다.

### 3.2 라우트 구조

- `/store/inbound` -> `/store/inbound/list`
- `/store/inbound/list`
- `/store/inbound/list/:id`
- `/store/inbound/history` -> `/store/inbound/list`
- `/store/inbound/history/:id` -> `/store/inbound/list/:id`
- `/store/inbound/analysis`

## 4. 화면 설계

### 4.1 입고 리스트

목적:

- 승인 완료 이후 입고가 진행 중인 발주와 입고 완료된 발주를 한 화면에서 조회한다.
- 배송 완료 상태의 처리 대상을 빠르게 식별하고 상세로 진입시킨다.

구성:

- 상태 탭: `전체 / 배송 준비중 / 배송 중 / 배송 완료 / 입고 완료`
- 검색: 발주번호, 상품명, 카테고리
- 기간 필터
- 정렬 옵션
- 상단 액션 배너
- 리스트 테이블

리스트 컬럼:

- 발주일시
- 발주번호
- 대표 상품명
- 총 SKU
- 입고 수량
- 입고 상태
- 입고 예정일
- 입고 확정 처리

행동 유도:

- `배송 완료` 상태 발주가 있으면 상단에 `입고 확정 필요` 배너를 노출한다.
- `배송 완료` 행의 `입고 확정 처리` 컬럼에는 `입고 확정하기` 버튼을 노출한다.
- `배송 준비중`, `배송 중`은 `대기`
- `입고 완료`는 `처리 완료`
- 버튼 클릭은 리스트 즉시 처리 대신 상세 페이지 이동으로 고정한다.

### 4.2 입고 상세

목적:

- 발주 원문과 입고 진행 상태를 함께 확인한다.
- 배송 완료 상태에서만 최종 입고 확정을 수행한다.

구성:

- 상단 헤더: 발주번호, 발주일시, 매장, 입고 상태 배지
- 상단 안내 배너: 배송 완료 상태일 때만 노출
- 좌측: 기본 정보, SKU 라인 테이블, 발주 메모
- 우측: 입고 정책, 입고 진행 이력, 입고 확정 영역

SKU 테이블 컬럼:

- 품목코드
- 상품명
- 옵션
- 카테고리
- 현재고
- 발주 수량

확정 전 안내:

- 실제 입고된 수량과 발주 수량을 확인한 뒤 확정을 진행하도록 안내한다.
- 입고 확정 시 확인된 수량이 매장 재고에 즉시 반영된다는 점을 명시한다.

입고 확정 조건:

- `status === APPROVED`
- `inboundStatus === ARRIVED`

### 4.3 입고 분석

목적:

- 입고 진행 현황을 요약 수치로 빠르게 파악한다.

MVP 범위:

- 입고 완료 건수
- 누적 입고 수량
- 배송 중 건수
- 배송 완료 건수
- 상태별 건수
- 카테고리별 입고 수량

## 5. 상태 모델

### 5.1 발주 상태

- `REQUESTED`: 승인 대기
- `APPROVED`: 승인 완료 후 입고 진행 중
- `COMPLETED`: 입고 확정까지 끝난 종료 발주
- `CANCELLED`: 취소

### 5.2 입고 상태

- `READY_TO_SHIP`: 배송 준비중
- `IN_TRANSIT`: 배송 중
- `ARRIVED`: 배송 완료
- `RECEIVED`: 입고 완료

### 5.3 상태 전환 규칙

- 발주 승인 시 `APPROVED + READY_TO_SHIP`
- 배송 출발 시 `IN_TRANSIT`
- 매장 도착 시 `ARRIVED`
- 매장 입고 확정 시 `COMPLETED + RECEIVED`

## 6. 데이터 모델

### 6.1 StoreOrder 확장 필드

- `inboundStatus`
- `inboundExpectedAt`
- `inboundCompletedAt`
- `inboundConfirmedBy`
- `inboundStatusHistory[]`

### 6.2 StoreOrderItem 입고 관련 필드

- `expectedInboundQuantity`

이번 버전에서는 `expectedInboundQuantity = requestedQuantity`를 기본으로 사용한다.

## 7. 연계 규칙

### 7.1 발주 연계

- 승인 완료된 발주만 입고 흐름 대상이다.
- 입고 완료 시 발주 상태를 `COMPLETED`로 변경한다.
- 발주 상세에서는 발주 상태와 입고 상태를 병행해 보여주되, 입고 상세는 입고 상태 중심으로 보여준다.

### 7.2 재고 연계

- 입고 확정 시 `inventory.receiveItems()`를 호출한다.
- 각 SKU의 `expectedInboundQuantity`만큼 매장 재고를 증가시킨다.
- 중복 입고 확정은 허용하지 않는다.

### 7.3 창고 연계 준비

- 창고 UI 구현 전이라도 아래 상태 전환 액션은 제공한다.
- `markReadyToShip(orderId)`
- `markInTransit(orderId)`
- `markArrived(orderId)`
- `confirmInbound(orderId, confirmedBy)`

## 8. 테스트 기준

- `APPROVED + READY_TO_SHIP` 발주가 입고 리스트에 노출되어야 한다.
- `APPROVED + ARRIVED` 발주에 `입고 확정하기` 액션이 보여야 한다.
- `COMPLETED + RECEIVED` 발주는 입고 완료 상태로 조회되어야 한다.
- 입고 확정 후 재고가 증가해야 한다.
- 입고 확정 후 발주 상태가 `COMPLETED`로 바뀌어야 한다.
- 입고 상태 이력에 `READY_TO_SHIP -> IN_TRANSIT -> ARRIVED -> RECEIVED`가 남아야 한다.
- `vite build`가 성공해야 한다.

## 9. 설계 메모

- 본 문서는 입고 기능의 기능/상태/흐름 기준을 정리한 문서다.
- 버튼 스타일, 배지 위치, 타임라인 애니메이션 등 세부 디자인 튜닝은 본 문서 범위에서 제외한다.

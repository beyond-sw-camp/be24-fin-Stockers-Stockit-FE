# 매장 발주영역 통합 설계안

## 1. 문서 목적

이 문서는 Stockit 기획서 v2.0 기준으로 매장관리자 발주 기능의 목표, 화면 구조, 상태 모델, 데이터 모델, 메뉴 구조, 구현 원칙을 통합 정리한 문서다.

현재 매장 발주 기능은 판매 기능과 동일하게 `요청 / 내역 / 분석` 구조로 분리하며, 매장 재고와 동일한 의류 SKU 도메인을 기준으로 동작한다. 본사 거래처 발주와는 별도 도메인으로 관리하고, 매장에서는 `매장 -> 창고 발주 요청` 흐름에 집중한다.

## 2. 범위와 기본 방향

### 2.1 업무 범위

- 매장 발주 요청 등록
- 매장 발주 요청 수정
- 매장 발주 요청 취소
- 매장 발주 내역 조회
- 매장 발주 상세 조회
- 매장 발주 분석 조회

### 2.2 이번 버전의 구현 범위

- 상태 범위는 `승인 대기`, `승인 완료`, `취소` 3개만 사용
- 자동 배치 승인 스케줄러는 구현하지 않고 정책 안내 문구만 노출
- 입고 관리와의 완전한 연결은 후속 범위로 남기고, 발주 도메인과 화면 구조를 먼저 안정화
- 분석은 MVP 수준의 집계형 화면으로 제한

### 2.3 설계 원칙

- 본사 `purchaseOrder` 스토어를 재사용하지 않고 매장 전용 `storeOrders` 스토어를 둔다
- 데이터 기준은 반드시 현재 판매/재고와 같은 의류 SKU 체계를 따른다
- 발주 요청은 상품 단위가 아니라 SKU 단위다
- 하나의 요청서에 여러 SKU를 담아 하나의 발주건으로 저장한다
- 수정과 취소는 `승인 대기` 상태에서만 허용한다

## 3. 메뉴 구조와 라우트

### 3.1 메뉴 구조

- 상위 메뉴: `발주 관리`
- 하위 메뉴: `발주 요청`
- 하위 메뉴: `발주 내역`
- 하위 메뉴: `발주 분석`

기존 매장 메뉴의 `주문 관리`는 `발주 관리`로 전환한 상태를 기준으로 한다.

### 3.2 라우트 구조

- `/store/orders` -> `/store/orders/request` 리다이렉트
- `/store/orders/request`
- `/store/orders/request/:id/edit`
- `/store/orders/history`
- `/store/orders/history/:id`
- `/store/orders/analysis`

## 4. 화면 설계

### 4.1 발주 요청

목적:

- 부족 재고 SKU를 빠르게 탐색하고 발주 요청서를 작성한다
- 여러 SKU를 하나의 발주건으로 묶어 저장한다

레이아웃:

- 좌측: `매장 발주 검색` 패널
- 우측: `발주 요청서` 패널

좌측 검색 패널 구성:

- 상품명 검색
- 대분류 필터
- 소분류 필터
- 색상 필터
- 사이즈 필터
- 정렬 필터
- SKU 테이블

정렬 옵션:

- `부족 SKU 우선` 기본값
- `카테고리순`
- `상품명순`
- `재고 부족한순`
- `재고 많은순`

대분류 필터/카테고리순 기준:

- `상의 -> 바지 -> 치마 -> 아우터`
- 가나다순이 아니라 기획 고정 순서를 사용

SKU 테이블 컬럼:

- 품목코드
- 상품명
- 옵션
- 카테고리
- 실재고
- 가용재고
- 안전 재고
- 권장 발주량
- 상태
- 추가

상태 표기:

- `품절`
- `부족`
- `정상`

상태 판단 기준:

- `품절`: 실재고 0
- `부족`: 안전재고 이하 또는 권장 발주량 존재
- `정상`: 그 외

요청서 패널 구성:

- 요청 SKU 카드 리스트
- 각 SKU별 품목코드, 상품명, 카테고리/옵션, 실재고, 가용재고, 안전재고, 권장 수량 표시
- 수량 증감 및 직접 입력
- 메모 입력
- 총 SKU
- 총 요청 수량
- 전체 비우기
- 발주 요청 등록 또는 수정 저장
- 운영 정책 안내

요청 화면 동작 규칙:

- 같은 SKU를 다시 담으면 신규 라인이 아니라 요청 수량만 누적
- 수량은 1 이상 정수만 허용
- 권장 발주량 기본 계산은 `max(0, safetyStock - currentStoreStock)`
- 가용재고는 `실재고 + 입고예정 수량`
- 수정 모드에서는 기존 요청건을 같은 요청 화면에 불러와 재사용
- 수정 가능 여부는 진입 시와 저장 시 모두 다시 검증

### 4.2 발주 내역

목적:

- 현재 및 과거 발주건을 목록으로 조회한다
- 목록에서 특정 발주건을 선택해 상세 페이지로 이동한다

레이아웃:

- 목록 전용 페이지
- 상세 패널을 같은 화면에 두지 않는다

조회 기능:

- 상태 탭
- 발주번호/상품명/카테고리 검색
- 기간 필터
- 정렬 옵션

목록 컬럼:

- 요청일시
- 발주번호
- 대표 상품명
- 총 SKU
- 총 요청 수량
- 상태

상태 탭:

- 전체
- 승인 대기
- 승인 완료
- 취소

### 4.3 발주 상세

목적:

- 발주 1건의 헤더 정보와 SKU 라인 정보를 상세 확인한다
- 승인 대기 상태일 때 수정/취소를 수행한다

레이아웃:

- 좌측: 상세 정보, SKU 라인 테이블, 메모, 취소 사유
- 우측: 정책 안내, 진행 이력, 액션 버튼

상세 정보:

- 발주번호
- 요청일시
- 요청자
- 매장
- 상태

상세 SKU 테이블 컬럼:

- 품목코드
- 상품명
- 옵션
- 카테고리
- 실재고
- 가용재고
- 요청 수량

우측 패널 구성:

- 정책 안내
- 진행 이력
- 수정 버튼
- 취소 버튼
- 처리 결과 메시지

수정/취소 규칙:

- `승인 대기` 상태에서만 버튼 노출
- 수정 버튼은 `/store/orders/request/:id/edit`로 이동
- 취소는 상세 페이지 내 확인 모달을 통해 처리

### 4.4 발주 분석

목적:

- 매장 발주 현황을 빠르게 파악한다

MVP 범위:

- 총 발주건 수
- 누적 요청 수량
- 상태별 건수
- SKU별 요청량 상위
- 카테고리별 요청 비중

카테고리 집계 정렬:

- 대분류 기준은 `상의 -> 바지 -> 치마 -> 아우터`
- 같은 대분류 안에서는 소분류 기준으로 정리

## 5. 상태 모델

### 5.1 내부 상태값

- `REQUESTED`
- `APPROVED`
- `CANCELLED`

### 5.2 UI 노출 문구

- `REQUESTED` -> `승인 대기`
- `APPROVED` -> `승인 완료`
- `CANCELLED` -> `취소`

### 5.3 상태 이력

모든 발주건은 상태 이력 배열을 가진다.

예시:

- 승인 대기 등록
- 승인 완료 처리
- 취소 처리

이번 버전에서는 승인 완료와 취소만 수동 데이터/시드 데이터로 표현하고, 실제 자동 상태 전환 로직은 포함하지 않는다.

## 7. 정렬과 우선순위 규칙

### 7.1 부족 SKU 우선

기본 정렬은 아래 순서를 따른다.

- 부족/안전재고 미만 SKU 우선
- 권장 발주량이 큰 SKU 우선
- 동일 조건이면 상품명순

### 7.2 카테고리순

- 대분류: `상의 -> 바지 -> 치마 -> 아우터`
- 소분류: 문자열 순
- 상품명: 문자열 순

### 7.3 재고 부족한순

- 실재고 오름차순
- 동률이면 권장 발주량 내림차순
- 동률이면 상품명순

### 7.4 재고 많은순

- 실재고 내림차순
- 동률이면 권장 발주량 내림차순
- 동률이면 상품명순

## 8. 현재 코드 기준 설계 반영 결과

현재 구현은 아래와 같이 설계에 맞춰 정리되어 있다.

- 매장 메뉴는 `발주 요청 / 발주 내역 / 발주 분석` 구조로 분리
- 발주 요청은 검색 패널 + 요청서 패널의 2단 구조
- 발주 내역은 목록 전용 페이지
- 발주 상세는 별도 페이지
- 발주 상태는 `승인 대기 / 승인 완료 / 취소`만 사용
- 품목코드, 실재고, 가용재고, 안전 재고, 권장 발주량을 모두 노출
- 대분류 순서와 카테고리 정렬 기준은 `상의 -> 바지 -> 치마 -> 아우터`

## 9. 후속 확장 과제

- 입고 관리 화면과 발주 상세 상태 연동
- 승인 완료 이후 출고/입고 단계 확장 여부 검토
- 권장 발주량 계산식 고도화
- 발주 분석 차트 고도화
- 본사/창고 시스템과의 실제 데이터 연계

## 10. 테스트 기준

- 여러 SKU를 한 요청서에 담아 1건의 발주로 저장할 수 있어야 한다
- 같은 SKU를 여러 번 담으면 요청 수량만 누적되어야 한다
- `승인 대기` 상태에서만 수정/취소가 가능해야 한다
- 발주 상세에서 수정 버튼이 요청 수정 화면으로 이동해야 한다
- 발주 상세에서 취소 시 상태와 이력이 갱신되어야 한다
- 발주 요청 화면에서 실재고와 가용재고가 함께 보여야 한다
- 대분류 필터와 카테고리순 정렬이 `상의 -> 바지 -> 치마 -> 아우터` 순서를 따라야 한다
- 재고 부족한순 / 재고 많은순 정렬이 정상 동작해야 한다
- `vite build`가 성공해야 한다

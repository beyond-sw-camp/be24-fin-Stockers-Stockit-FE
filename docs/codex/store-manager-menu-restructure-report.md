# 매장 관리자 메뉴 구조 개편 및 구현 보고서

> **프로젝트**: StockIT ERP Frontend  
> **작성일**: 2026-04-20  
> **대상 권한**: 매장 관리자  
> **구현 범위**: 매장 관리자 탑바/사이드바 메뉴 구조 개편, 대시보드/통계 화면 추가, 주문 요청 플로우 구현  
> **검증 상태**: `npm.cmd run build` 성공

---

## 1. 개요

이번 작업은 매장 관리자가 사용하는 화면의 정보 구조를 실제 업무 흐름에 맞게 재정리하는 것을 목표로 했다.

기존 구조에서는 `roleMenus.store` 하나가 탑바와 사이드바에 동시에 사용되고 있어, 탑바가 대분류 역할을 하지 못하고 사이드바도 세부 기능을 표현하지 못했다. 또한 매장 관리자의 주문 업무에서 `주문`, `발주` 용어가 섞일 가능성이 있어 업무 정의를 명확히 했다.

이번 개편에서 정한 핵심 정의는 다음과 같다.

| 용어 | 정의 |
| --- | --- |
| 주문 / 주문 요청 | 매장 점주가 본사 또는 물류창고에 필요한 물품을 요청하는 행위 |
| 발주 | 본사가 거래처 또는 공장에 필요한 물품을 요청하는 행위 |

따라서 매장 관리자 화면에서는 `발주`라는 표현을 사용하지 않고, `주문` 또는 `주문 요청`으로 용어를 통일했다.

## 2. 최종 메뉴 구조

### 2.1 탑바 메뉴

매장 관리자 탑바는 큰 업무 단위로 구성했다. 최종 순서는 사용자가 요청한 대로 `판매`를 `재고 관리`보다 앞에 배치했다.

| 순서 | 탑바 메뉴 | 라우트 | 목적 |
| --- | --- | --- | --- |
| 1 | 대시보드 | `/store/dashboard` | 매장 운영 요약 확인 |
| 2 | 판매 | `/store/pos` | 판매 처리 및 재고 차감 |
| 3 | 재고 관리 | `/store/inventory` | 매장 보유 재고, 품절/부족 재고 확인 |
| 4 | 주문 관리 | `/store/orders` | 본사/물류창고로 주문 요청 및 주문 내역 관리 |
| 5 | 입고 관리 | `/store/inbound` | 입고 예정, 검수, 확정, 과거 이력 확인 |
| 6 | 통계/모니터링 | `/store/monitoring` | 핵심 지표 및 자연어 질의 |

### 2.2 사이드바 메뉴

사이드바는 선택된 탑바 메뉴에 따라 세부 기능만 보여주도록 분리했다.

| 탑바 메뉴 | 사이드바 메뉴 |
| --- | --- |
| 대시보드 | 매장 요약, 오늘 알림 |
| 판매 | 판매 페이지 |
| 재고 관리 | 매장 보유 재고 조회, 품절/부족 재고 |
| 주문 관리 | 주문 요청, 주문 목록 조회 |
| 입고 관리 | 입고 리스트 조회, 입고 검수, 입고 확정 처리, 과거 입고 이력 조회 |
| 통계/모니터링 | 핵심 지표, 자연어 맞춤 질의 |

### 2.3 제외한 메뉴

사용자와 논의한 결과, 이번 개편에서는 아래 메뉴를 제외했다.

| 제외 메뉴 | 제외 이유 |
| --- | --- |
| 재고 변동 이력 | 현재 매장 관리자 1차 메뉴에는 불필요하다고 판단 |
| 재고 조정 요청 | 현재 요구 범위에서 제외 |
| 판매 내역 조회 | 판매 화면 외 세부 판매 메뉴는 우선 제외 |
| 환불/취소 | 판매 탑바 하위 메뉴에서는 제외 |
| 판매 재고 차감 내역 | 판매 페이지 자체에서 재고 차감 흐름을 처리 |
| 시스템 권장 주문량 | 별도 사이드바 메뉴가 아니라 주문 요청 화면 내부 카드로 처리 |
| 주문 상세 조회 | 주문 목록에서 선택해 진입하는 상세 영역으로 처리 |
| 주문 수정/취소 | 주문 상세 영역 안에서 처리 |

## 3. 주문 관리 UX 플로우

### 3.1 주문 요청 화면

`주문 요청`은 매장 점주가 본사/물류창고에 물품을 요청하는 핵심 화면이다. 별도 메뉴를 여러 개로 쪼개지 않고, 하나의 흐름 안에서 작업이 이어지도록 구현했다.

최종 흐름은 다음과 같다.

```text
상품 검색
  -> 추천 주문량 카드 확인
  -> 상품을 주문 장바구니에 담기
  -> 주문 요청서 작성/확인
  -> 요청 수량, 희망 입고일, 메모 수정
  -> 최종 주문 요청
  -> 주문 목록 조회로 이동
```

### 3.2 추천 주문량 카드

`시스템 권장 주문량`은 사이드바 메뉴로 만들지 않고 `주문 요청` 화면의 보조 영역에 카드 형태로 배치했다.

카드에 표시되는 정보는 다음과 같다.

| 항목 | 설명 |
| --- | --- |
| 추천 품목 | 주문 요청이 필요한 상품명 |
| 현재 재고 | 매장 현재 보유 수량 |
| 안전 재고 | 기준 재고 수량 |
| 추천 주문량 | 시스템이 제안하는 주문 수량 |
| 예상 소진일 | 현재 판매 흐름 기준 예상 소진 시점 |

추천 주문량 카드를 클릭하면 해당 품목이 추천 수량과 함께 주문 장바구니에 자동으로 담긴다.

### 3.3 주문 장바구니

상품 검색 결과에서 `장바구니 담기`를 누르거나 추천 주문량 카드를 클릭하면 주문 장바구니에 품목이 추가된다.

장바구니에서 가능한 작업은 다음과 같다.

| 기능 | 설명 |
| --- | --- |
| 수량 수정 | 품목별 주문 요청 수량을 수정 |
| 품목 삭제 | 장바구니에서 특정 품목 제거 |
| 예상 금액 확인 | 수량과 단가 기준 총 금액 확인 |
| 주문 요청서 작성 | 장바구니 품목을 기반으로 주문 요청서 섹션 생성 |

### 3.4 주문 요청서

장바구니에 담긴 품목을 바로 제출하지 않고, 먼저 주문 요청서를 생성하도록 구현했다.

주문 요청서에서 가능한 작업은 다음과 같다.

| 기능 | 설명 |
| --- | --- |
| 품목별 요청 수량 수정 | 장바구니에서 넘어온 수량을 최종 제출 전 수정 |
| 희망 입고일 입력 | 매장이 원하는 입고 날짜 지정 |
| 요청 메모 입력 | 요청 사유 또는 참고 사항 작성 |
| 총 요청 금액 확인 | 품목별 단가와 수량 기준 총액 확인 |
| 최종 주문 요청 | 주문 상태 `대기`로 주문 요청 생성 |

최종 주문 요청이 완료되면 새 주문이 주문 목록 맨 위에 추가되고, 화면은 `주문 목록 조회`로 전환된다.

### 3.5 주문 목록 및 상세

`주문 목록 조회`에서는 매장 점주가 요청한 주문 내역 리스트를 보여준다.

목록에 표시되는 정보는 다음과 같다.

| 항목 | 설명 |
| --- | --- |
| 주문 번호 | 주문 요청 식별 번호 |
| 요청일 | 주문 요청 생성 시각 |
| 대표 품목 | 주문 품목 중 첫 번째 품목명 |
| 품목 수 | 주문에 포함된 품목 개수 |
| 상태 | 대기, 승인, 배송 준비, 취소 |
| 희망 입고일 | 주문 요청서에서 지정한 날짜 |

주문 목록의 행을 클릭하면 오른쪽 상세 패널에서 상세 내역을 조회한다. 주문 상세는 별도 사이드바 메뉴로 만들지 않았다.

### 3.6 주문 수정/취소

주문 수정과 취소는 `주문 상세 내역` 안에서 처리한다.

수정/취소 가능 조건은 다음과 같다.

| 주문 상태 | 수정 가능 | 취소 가능 | 설명 |
| --- | --- | --- | --- |
| 대기 | 가능 | 가능 | 본사 또는 물류창고가 아직 처리하지 않은 상태 |
| 승인 | 불가 | 불가 | 이미 승인된 주문 |
| 배송 준비 | 불가 | 불가 | 물류 처리 단계 진입 |
| 취소 | 불가 | 불가 | 이미 취소된 주문 |

`대기` 상태 주문에서만 `수정`, `주문 취소` 버튼이 활성화된다.

## 4. 구현 내용

### 4.1 메뉴 설정 구조 변경

변경 파일:

- `src/config/roleMenus.js`

주요 변경 사항:

- 기존 `roleMenus.store`는 탑바 대분류 메뉴로 재정의했다.
- 매장 관리자 전용 `storeTopMenus`를 추가했다.
- 탑바 메뉴별 사이드바 매핑을 위한 `storeSideMenusByTopMenu`를 추가했다.
- 매장 관리자 홈 경로를 `/store/dashboard`로 변경했다.
- 권한 표시명 한글 깨짐을 정상화했다.

핵심 구조:

```js
export const storeTopMenus = roleMenus.store

export const storeSideMenusByTopMenu = {
  대시보드: [...],
  판매: [...],
  '재고 관리': [...],
  '주문 관리': [...],
  '입고 관리': [...],
  '통계/모니터링': [...],
}
```

### 4.2 라우터 변경

변경 파일:

- `src/router/index.js`

추가된 라우트:

| 라우트 | 컴포넌트 | 설명 |
| --- | --- | --- |
| `/store/dashboard` | `StoreDashboardView.vue` | 매장 관리자 메인 대시보드 |

기존 매장 관리자 라우트도 새 메뉴 의미에 맞춰 유지했다.

| 라우트 | 역할 |
| --- | --- |
| `/store/pos` | 판매 |
| `/store/inventory` | 재고 관리 |
| `/store/orders` | 주문 관리 |
| `/store/inbound` | 입고 관리 |

### 4.3 공통 레이아웃 변경

변경 파일:

- `src/components/common/AppLayout.vue`

주요 변경 사항:

- 탑바 메뉴와 사이드바 메뉴를 별도 props로 받아 표시하도록 정리했다.
- 탑바 클릭 시 `menu.path` 기준으로 라우터 이동을 수행한다.
- 사이드바 클릭 시 `item.path`가 있으면 라우터 이동, 없으면 `activeSideMenu`만 갱신한다.
- 기존 직접 구현 SVG 아이콘 일부를 `lucide-vue-next` 아이콘으로 정리했다.
- 깨진 한글 문자열을 정상화했다.
- `showSystemCard`가 필요한 화면에서는 사이드바 하단에 시스템 상태를 표시하도록 유지했다.

## 7. 현재 구현의 한계와 후속 작업 제안

### 7.1 현재 한계

이번 구현은 프론트엔드 UI와 로컬 상태 중심의 MVP 성격이다.

| 한계 | 설명 |
| --- | --- |
| 주문 데이터 영속성 없음 | 주문 요청 목록은 컴포넌트 내부 `ref` 상태로 관리되어 새로고침 시 초기화됨 |
| 실제 백엔드 연동 없음 | 주문 생성, 수정, 취소, 상세 조회가 API와 연결되어 있지 않음 |
| 주문 상태 자동 전이 없음 | 승인, 배송 준비 등 상태 변화는 샘플 데이터로만 표현 |
| 자연어 질의 실행 없음 | 질의 UI만 있고 실제 AI/검색 결과는 연결되어 있지 않음 |
| 입고 확정 처리 실제 동작 없음 | 입고 목록 필터링 중심이며 확정 mutation은 미구현 |

### 7.2 후속 작업 제안

다음 단계에서 진행하면 좋은 작업은 다음과 같다.

| 우선순위 | 작업 | 설명 |
| --- | --- | --- |
| 높음 | 주문 요청 Pinia store 분리 | `StoreOrdersView.vue` 내부 주문 상태를 `orders` store로 분리 |
| 높음 | 주문 API 연동 | 주문 생성, 목록, 상세, 수정, 취소 API 연결 |
| 중간 | 주문 요청서 라우트 분리 | 필요 시 `/store/orders/request-form` 또는 query state로 분리 |
| 중간 | 입고 확정 실제 처리 | 입고 확정 시 재고 증가와 입고 상태 변경 |
| 중간 | 대시보드 데이터 연결 | 판매, 재고, 입고, 주문 데이터를 실제 store/API 기반으로 집계 |
| 낮음 | 자연어 질의 기능 연동 | 통계/모니터링의 질의 실행 기능 구현 |

## 8. 최종 정리

이번 작업으로 매장 관리자 화면은 기존의 단순 메뉴 나열 구조에서 벗어나, 탑바는 큰 업무 단위, 사이드바는 선택된 업무의 세부 기능을 보여주는 구조로 바뀌었다.

특히 주문 관리 영역은 사용자가 정의한 업무 용어에 맞춰 `발주`가 아닌 `주문 요청` 중심으로 정리되었고, 실제 매장 점주가 사용할 법한 흐름인 `상품 검색 → 추천 주문량 반영 → 장바구니 → 주문 요청서 → 최종 요청 → 목록/상세/수정/취소`까지 한 화면에서 동작하도록 구현되었다.

빌드 검증까지 완료되어 현재 구현은 프론트엔드 기준으로 동작 가능한 상태다.

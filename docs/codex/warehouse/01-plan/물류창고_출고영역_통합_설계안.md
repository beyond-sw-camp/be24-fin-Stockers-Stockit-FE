# 물류창고 출고영역 통합 설계안

## 1. 문서 목적

본 문서는 물류창고 관리자 관점에서 출고 관리 화면을 `매장 출고`, `창고간 이동`, `순환재고 판매 출고`까지 통합해 운영하기 위한 기준을 정리한 설계 문서다.

이번 단계는 프론트 통합 구조를 먼저 완성하고, 백엔드 API는 후속 단계에서 최종 계약을 고정한다.

## 2. 범위와 기본 방향

### 2.1 업무 범위

- 통합 출고 리스트 조회
- 출고 상세 조회
- 출고 진행 이력 조회
- 출고확정 처리(매장 출고 유형만)
- 완료 이력 조회

### 2.2 이번 버전 구현 범위

- 3유형 통합 조회를 지원한다.
  - `STORE_OUTBOUND`
  - `WH_TRANSFER`
  - `CIRCULAR_SALE`
- 상태 모델은 공통 3단계만 사용한다.
  - `READY_TO_SHIP`
  - `IN_TRANSIT`
  - `COMPLETED`
- 상태 변경 액션은 `STORE_OUTBOUND`만 활성화한다.
- `WH_TRANSFER`, `CIRCULAR_SALE`는 조회 전용으로 제공한다.

### 2.3 설계 원칙

- `warehouseOutbound`를 단일 읽기 모델로 사용한다.
- source adapter 합성 구조로 구현한다.
  - `fromStoreOrders()`
  - `fromWarehouseTransfersMock()`
  - `fromCircularSalesMock()`
- `storeOrders`와 충돌 시 원본 상태(`inboundStatus`)를 우선한다.

## 3. 메뉴 구조와 라우트

- 상위 메뉴: `입/출고 관리`
- 하위 메뉴: `출고 관리`

라우트:

- `/warehouse/outbound`
- `/warehouse/outbound/:id`

## 4. 화면 설계

### 4.1 출고 리스트

- 유형 필터: `전체 / 매장 출고 / 창고간 이동 / 순환재고 판매`
- 상태 탭: `전체 / 출고 준비중 / 배송중 / 완료`
- 기간 필터: `전체 / 일 / 월 / 년` + 직접 날짜 입력
- 검색: `출고번호, 발주번호, 출고처, 목적지, 대표상품`
- 목록 컬럼:
  - 출고번호
  - 발주번호
  - 유형
  - 출고처
  - 목적지
  - 대표상품
  - 총 SKU
  - 총 수량
  - 상태
  - 요청일시

### 4.2 출고 상세

- 2단 패널 유지: 좌 70 / 우 30
- 우측 상단 상태 배지 + 진행 이력
- 액션 영역:
  - `STORE_OUTBOUND`: 출고확정 버튼 활성
  - `WH_TRANSFER`, `CIRCULAR_SALE`: 조회 전용 안내

## 5. 통합 데이터 모델

### 5.1 WarehouseOutbound

- `outboundId`
- `orderId`
- `outboundType` (`STORE_OUTBOUND | WH_TRANSFER | CIRCULAR_SALE`)
- `sourceType`
- `targetType`
- `sourceName`
- `targetName`
- `status`
- `requestedAt`
- `confirmedAt`
- `confirmedBy`
- `completedAt`
- `headlineProduct`
- `totalSkuCount`
- `totalRequestedQuantity`
- `items[]`
- `outboundStatusHistory[]`
- `actionable`

### 5.2 상태 매핑 규칙(매장 출고)

- `READY_TO_SHIP` <- `storeOrders.inboundStatus === READY_TO_SHIP`
- `IN_TRANSIT` <- `storeOrders.inboundStatus === IN_TRANSIT | ARRIVED`
- `COMPLETED` <- `storeOrders.inboundStatus === RECEIVED` 또는 `storeOrders.status === COMPLETED`

## 6. 스토어 액션

- `getOutboundList(filters)`
- `getOutboundById(outboundId)`
- `confirmOutbound(outboundId, actor)`
- `getCompletedHistory(filters)`

## 7. API 계약 초안(후속 확정)

- 통합 목록 API 응답은 `WarehouseOutbound[]` 기준으로 정규화한다.
- 필수 필드: `outboundId`, `outboundType`, `status`, `requestedAt`, `sourceName`, `targetName`, `items`.
- 상태 전이 API는 유형별 지원 범위를 분리한다.
  - 1차: `STORE_OUTBOUND`만 출고확정(배송중 전환)
  - 후속: `WH_TRANSFER`, `CIRCULAR_SALE` 전이 액션 확장

## 8. 테스트 기준

- 3유형 데이터가 하나의 리스트로 조회되어야 한다.
- 유형/상태/기간/검색 조합 필터가 일관되게 동작해야 한다.
- `confirmOutbound()`는 `STORE_OUTBOUND`만 성공하고 나머지는 조회 전용으로 차단되어야 한다.
- 매장 입고 화면(`store/inbound`)의 기존 상태 흐름이 유지되어야 한다.
- `vite build`가 성공해야 한다.

기준 문서:

- 기획서: `C:\Users\saral\Desktop\Stockit\기획\Stockit_기획서 v2.0.pdf`
- 현재 코드: `C:\Users\saral\Desktop\Stockit\stockit-fe`

## 1. 총평

현재 프론트엔드는 역할 기반 재고관리 ERP의 형태를 이미 갖추고 있으며, 본사/매장/창고 분리, POS, 발주, AI 리포트, ESG 대시보드까지 화면 범위가 넓다.  
하지만 기획서 기준으로 보면 현재 코드는 `데모형 업무 UI` 수준에 가깝고, 도메인 일관성 및 핵심 차별 기능의 구현 깊이는 아직 기획 수준에 못 미친다.

가장 큰 갭은 아래 3가지다.

- 도메인이 `패션/의류`보다 `잡화/전자제품/사무용품` 쪽으로 섞여 있음
- 순환 재고 시스템이 기획서 수준으로 구현되지 않음
- 실제 운영 흐름보다 더미 스토어 중심의 독립 화면이 많음

## 2. 맞아 있는 부분

기획서와 현재 코드가 잘 맞는 부분도 분명하다.

### 2.1 역할 기반 구조

- `src/router/index.js`에서 `hq`, `store`, `warehouse` 역할 라우팅 분리
- `src/stores/auth.js`에서 역할 기반 로그인 처리
- `src/config/roleMenus.js`에서 역할별 메뉴 분기

이는 기획서의 `CentralManager / StoreManager / WarehouseManager` 구조와 방향이 일치한다.

### 2.2 본사/매장/창고 화면 분리

- `src/views/hq/*`
- `src/views/store/*`
- `src/views/warehouse/*`

기획서의 3개 사용자군을 UI 상으로 나눈 점은 적절하다.

### 2.3 일부 핵심 기능의 프로토타입 존재

- 매장 판매 및 재고 차감: `src/views/store/StorePosView.vue`, `src/stores/inventory.js`
- 발주 관리: `src/stores/purchaseOrder.js`, `src/views/hq/HqPurchaseOrderView.vue`
- 본사 AI 리포트: `src/stores/hqAiReport.js`, `src/views/hq/HqAiReportView.vue`
- 매장 AI 리포트: `src/stores/storeAiReport.js`
- ESG 대시보드: `src/stores/esg.js`, `src/views/hq/esg/EsgDashBoardView.vue`

즉, 기획서 핵심 차별 기능의 싹은 이미 코드에 들어와 있다.

## 3. 핵심 갭

## 3.1 도메인 불일치

기획서는 SPA 패션/의류 기업을 전제로 한다. 그러나 현재 코드는 아래처럼 도메인이 섞여 있다.

- `src/stores/inventory.js`: 충전기, 마우스, 복사용지, 가글, 머그컵
- `src/stores/purchaseOrder.js`: 사무용품, 전자제품, 생활용품 중심
- `src/stores/aiOrderRecommendation.js`: 전자/생활/사무 품목 중심
- `src/stores/hqAiReport.js`, `src/views/hq/esg/EsgDashBoardView.vue`: 의류/소재 중심 데이터 사용

즉, `재고/발주 영역`과 `AI/ESG 영역`의 도메인 모델이 서로 다르다.  
기획서 기준으로는 전반을 의류 중심으로 통일해야 한다.

영향:

- 순환 재고 판단 로직을 붙이기 어렵다.
- 소재/무게/사이즈/컬러 같은 의류 특성이 재고 모델에 없다.
- ESG 점수와 순환 재고가 실제 재고/발주 데이터와 연결되지 않는다.

## 3.2 인증/보안 수준 부족

기획서는 JWT 기반 인증/인가와 RBAC를 요구한다.  
현재는 `src/stores/auth.js`에서 더미 계정과 `localStorage`만 사용한다.

현재 상태:

- 더미 계정 하드코딩
- 토큰 없음
- 세션 만료/갱신 없음
- 사용자/권한 정보가 프론트 로컬 상태에만 존재

이는 MVP 데모 용도로는 괜찮지만, 기획서 기준 운영 구조와는 차이가 크다.

## 3.3 순환 재고 시스템의 부재

기획서의 핵심 차별화 1순위는 `순환 재고 시스템`이다.  
현재 코드에는 관련 화면 이름은 일부 보이지만, 기획서 수준의 실제 흐름은 아직 부족하다.

현재 확인된 요소:

- `src/views/hq/HqCircularInventoryCandidateView.vue`
- `src/views/hq/HqCircularInventoryView.vue`

추정 가능한 수준:

- 화면 진입점은 있으나, 기획서의 핵심 판정 기준인 장기 미판매, 판매율 저조, 사이즈 파편화, 소재/무게 기준 재분류가 전면적으로 연결되어 있지 않다.

기획서 대비 빠진 핵심:

- 입고 후 24개월 이상 장기 정체 기준
- 시즌 목표 대비 판매율 저조 판정
- 사이즈/컬러 편중 재고 판정
- 정상 판매 재고와 순환 재고의 성격 분리
- 소재/함량/총 무게 기준 검색
- 순환 거래처 판매 등록

## 3.4 창고 운영 흐름의 깊이 부족

기획서의 창고 관리자 기능은 WMS 흐름을 꽤 깊게 요구한다.

- 협력사 입고
- 피킹 리스트
- 적재 체크
- 배송 출발
- 공간 점유율
- 입출고 작업 진척도

현재 코드에는 창고 화면이 존재하지만, 스토어나 문서상으로는 실제 WMS 오퍼레이션 상태 머신이 충분히 보이지 않는다.

갭 포인트:

- 피킹 리스트 자동 생성 로직 부재
- 피킹 완료/상차 완료/배송 출발 상태 흐름 약함
- 창고 공간 점유율 계산 모델 부재
- 기사 매핑/배송 리스트 부재

## 3.5 본사 모니터링과 알림 체계의 미완성

기획서에서 본사는 전사 재고 현황판, 부족 알림, 공지, 발주량-가용량 비교, 창고 간 이동 지시를 수행해야 한다.

현재 코드에는 대시보드와 일부 모니터링 화면이 있으나 다음은 부족하거나 분리돼 있다.

- 재고 부족 자동 알림의 규칙 기반 동작
- 공지 관리
- 전국 매장 발주량 vs 창고 가용량 비교 뷰
- 창고 간 이동을 실제 재고 격리/운송/도착 확인 상태로 추적하는 흐름

## 3.6 AI 리포트의 데이터 연계 부족

기획서의 AI 리포트는 운영 데이터에서 중요한 신호를 추려 액션 브리핑을 제공해야 한다.

현재 장점:

- `src/stores/hqAiReport.js`
- `src/stores/storeAiReport.js`
- 수요 예측, 추천, 액션 목록 UI는 비교적 잘 구성돼 있음

하지만 현재는 대체로 랜덤/시드 데이터 기반이다.

갭 포인트:

- 실제 판매/재고/발주 데이터와의 연결 약함
- 기획서의 `진열 추천`, `연관 상품 추천`, `피크 방문 시간대` 등이 일부만 반영됨
- 본사 리포트와 매장 리포트가 실재 운영 이벤트에서 자동 생성되는 구조는 아님

## 3.7 ESG 현황판과 실재 데이터 연결 부족

현재 ESG 화면은 시각적으로 강하고 방향도 기획서와 잘 맞는다.  
하지만 아직 실제 순환 재고 처리량과 연결된 정량 시스템은 아니다.

갭 포인트:

- 재활용 의류 총 무게와 소재별 탄소 환산 계수의 실재 입력 연결 부족
- 순환 재고 처리 이벤트가 ESG 점수로 적립되는 체인 부재
- 나무 10단계 성장 로드맵이 실재 재고 순환 이벤트와 강하게 연결되지 않음

## 3.8 API/백엔드 연동 전제 부족

기획서는 ERP 연동형 MSA 시스템을 전제로 한다.  
현재는 `src/api/purchaseOrder.js`, `src/api/esg.js` 등이 주석 스텁 상태다.

의미:

- API 계약은 아직 실질적으로 확정되지 않음
- 프론트 스토어가 API 레이어를 대체하고 있음
- 실제 BE 연동 이후 구조 재정리가 필요할 가능성이 높음

## 4. 우선순위별 갭 정리

### 4.1 가장 먼저 맞춰야 할 것

- 제품/재고/발주 데이터 모델을 의류 도메인으로 통일
- 핵심 엔티티에 `카테고리, 시즌, 컬러, 사이즈, 소재, 무게` 추가
- 순환 재고 후보 판정에 필요한 필드 구조 선반영

### 4.2 MVP 범위에서 바로 중요한 것

- 매장 발주 요청-수정-취소-상태조회 흐름 정교화
- 창고 입고/피킹/출고 상태 흐름 정교화
- 본사의 전사 재고 및 창고 간 이동 흐름 구조화

### 4.3 기획서 차별화에 직접 연결되는 것

- 순환 재고 후보 판정 로직
- 소재/무게 기반 순환 재고 관리
- ESG 점수 적립과 순환 이벤트 연결
- AI 리포트와 실제 운영 데이터 연결

## 5. 결론

현재 프론트엔드는 기획서 방향과 완전히 어긋난 것은 아니다. 오히려 역할 기반 ERP, AI 리포트, ESG 대시보드까지 이미 들어와 있어 출발점은 좋다.  
하지만 지금 상태를 기획서 기준으로 보면 `업무 흐름 데모 + 화면 실험` 단계에 가깝고, `패션/의류 도메인 통일`과 `순환 재고 중심 구조 재정렬`이 꼭 필요하다.

앞으로의 작업 기준은 아래처럼 잡는 것이 적절하다.

- 기능 추가보다 먼저 도메인 정합성 확보
- 의류 재고 라이프사이클을 기준으로 화면과 스토어 재정렬
- 순환 재고, ESG, AI 리포트를 실제 재고 데이터 흐름과 연결

### 3.2 현재 구조의 문제점
- 출고/입고 상태 전이가 아직 로컬 스토어 더미 로직 중심이다.
  - 출고확정: `warehouseOutboundStore.confirmOutbound(...)`
  - 입고확정: `storeOrders.confirmInbound(...)`
- BE 기준으로 완료된 API(warehouse outbound, store inbound)와 1:1 연결되는 FE API 모듈이 부족하다.
- 상태 반영은 스토어 내부 가공 기준이어서, BE 단일 진실원과 정합성 확보가 필요하다.

### 3.3 이미 활용 가능한 기반
- `apiClient`, `unwrap`, `extractErrorMessage`가 이미 존재한다.
- 화면에서 버튼 disabled 처리 및 상태 라벨링 패턴이 이미 있으므로, UI 큰 변경 없이 연동 가능하다.

## 4. BE 연동 대상 (요청 기준)

### 4.1 창고 출고
- `GET /api/warehouse/outbound`
- `GET /api/warehouse/outbound/{outboundNo}`
- `POST /api/warehouse/outbound/{outboundNo}/confirm`
- `POST /api/warehouse/outbound/{outboundNo}/arrive`

### 4.2 매장 입고
- `GET /api/store/inbound`
- `GET /api/store/inbound/{inboundNo}`
- `POST /api/store/inbound/{inboundNo}/confirm`

### 4.3 발주 화면 보강
- 발주 조회 시 `inboundProgress`(n/m) 누락되면 표시 보강

## 5. 작업 설계

### 5.1 API 계층 설계
- 신규 API 모듈 추가(예정)
  - `src/api/warehouse/outbound.js`
  - `src/api/store/inbound.js`
- 함수 구성(예정)
  - outbound: `list`, `detail`, `confirm`, `arrive`
  - inbound: `list`, `detail`, `confirm`
- 모두 `unwrap` 반환 구조 통일

### 5.2 화면 연동 설계
- 출고 목록/상세
  - 초기 로드/필터 변경 시 목록 API 호출
  - 상세 진입 시 상세 API 호출
  - `confirm` 성공 후 상세/목록 재조회
  - `arrive`는 `IN_TRANSIT` 상태에서만 노출/활성
- 입고 목록/상세
  - 초기 로드/필터 변경 시 목록 API 호출
  - 상세 진입 시 상세 API 호출
  - `confirm` 성공 후 상세/목록 재조회
- 발주 조회
  - 응답 매핑 시 `inboundProgress` 우선 표시
  - 누락 시 수신/전체 수량(혹은 건수) 기반 fallback 문자열 생성

### 5.3 상태 반영 및 중복 클릭 방지
- 액션 버튼 호출 중 로딩 상태값으로 `disabled`
- 성공 시 즉시 반영 방식은 안정성 우선으로 `재조회` 사용
- 필요 시 상세 화면에 성공 메시지(토스트/배너) 노출

### 5.4 에러 처리 설계
- 모든 액션/조회 `try-catch` 처리
- 사용자 메시지 우선순위
  - BE 응답 `message`
  - HTTP 코드별 기본 메시지
  - 네트워크/타임아웃 메시지
- 공통 추출 함수 `extractErrorMessage` 우선 활용

## 6. 브랜치 전략 결론 (사용자 협의 완료)

### 6.1 권장 분리
- 브랜치 A: `출/입고 연동` 통합
  - 이유: 상태머신(`READY_TO_SHIP → IN_TRANSIT → ARRIVED → RECEIVED`)이 직접 연결되어 분리 시 중복/충돌 위험 증가
- 브랜치 B: `발주 inboundProgress 표시 보강`

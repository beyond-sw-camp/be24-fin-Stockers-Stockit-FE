# 순환재고 판매 리팩토링(S1~S3) 설계 및 구현 종합 보고서

## 1. 작업 개요
본 보고서는 순환재고 판매 FE-BE 연동 정합성 개선과 더미/파생 필드 최소화, 내역 화면 정합성 보정 작업(S1~S3)의 설계/구현 결과를 정리한다.

- 범위
  - S1: 구조 정리 및 리팩토링 방향 수립
  - S2: 판매 필드 정리(중복/파생 최소화)
  - S3: 판매 내역 화면 정합성 수정(산업군 복원 + 상태 컬럼 신설)
- 원칙
  - BE 응답 단일 소스 우선
  - FE 파생 필드 최소화
  - 화면 표기 의미와 바인딩 필드 일치

---

## 2. 단계별 설계 및 구현

## 2.1 S1 (설계 단계)
- 목표
  - `circularStock` 비대 원인 파악
  - 판매/재고 책임 분리 방향 정립
  - FE-BE 필드 표준화 기준 합의
- 핵심 설계 결정
  - 판매 데이터는 API 응답 중심으로 전환
  - Store에는 상태와 API 호출/상태 전이 로직 중심 유지
  - 뷰 전용 표시 계산은 뷰/composable로 이동
  - 필드 표준 키: `saleId`, `saleNo`, `status`, `outboundStatus`, `totalSoldQuantity`, `totalAmount`

## 2.2 S2 (필드 정리 단계)
- 목표
  - 판매 상세/목록에서 의미 중복 필드 제거
  - 단일 의미-단일 필드 원칙 적용
- 구현 요약
  - 상세/아이템 바인딩에서 중복 파생 필드 의존 축소
  - `productName` 단일 표준 사용, `itemName` 의존 제거 방향 반영
  - 합계 UI는 단일 총수량/총금액 중심으로 단순화
- 효과
  - 필드 혼동 감소
  - 유지보수 시 매핑 추적 비용 감소

## 2.3 S3 (내역 정합성 단계)
### A. BE 계약 확장
- 파일
  - `Stockit-BE/src/main/java/org/example/stockitbe/hq/circularsale/model/dto/CircularSaleDto.java`
  - `Stockit-BE/src/main/java/org/example/stockitbe/hq/circularsale/CircularSaleService.java`
- 변경
  - `ListRowRes`에 `buyerIndustryGroup` 추가
  - 목록 매핑 시 `buyer.getIndustryGroup()` 세팅

### B. FE 목록 매핑 반영
- 파일
  - `Stockit-FE/src/stores/hq/circularStock/circularStock.js`
- 변경
  - `mapSaleListFromApi`에 `buyerIndustryGroup` 매핑 추가

### C. 내역 화면 컬럼 정합성 보정
- 파일
  - `Stockit-FE/src/views/hq/circular-stock/HqCircularStockSalesHistoryView.vue`
- 변경
  - 기존 산업군 컬럼의 `status/outboundStatus` 결합 표기 제거
  - 산업군 컬럼을 `buyerIndustryGroup` 기반으로 복원
  - 마지막 컬럼에 상태 컬럼 신설
  - 상태 표기 기준을 `outboundStatus` 단일로 고정
  - 상태 한글 라벨/배지 UI 적용
    - `READY_TO_SHIP` → `출고 준비 중`
    - `IN_TRANSIT` → `배송중`
    - `ARRIVED` → `배송 완료`
    - 빈 값 → `-`
  - 상태 컬럼 정렬: 헤더/셀 모두 가운데 정렬

## 7. 결론
S1~S3 범위에서 판매 내역의 필드 의미 불일치 문제를 해소했고, 산업군/상태 표기를 도메인 의도에 맞게 정렬했다. 특히 상태는 `outboundStatus` 단일 정책과 배지 UI로 통일되어 사용자 해석 오류를 줄이는 방향으로 개선되었다.
